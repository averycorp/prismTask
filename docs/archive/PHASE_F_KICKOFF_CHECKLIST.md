# Phase F Kickoff Checklist (BetaTesting platform)

**Target kickoff:** 2026-05-15.
**Owner:** Avery (manual external verification — items below are not
self-verifiable from inside the repo).

This is the operational checklist for the launch derived from the
`D2_AND_PHASE_F_PREP_MEGA_AUDIT.md` item 5 audit. The audit confirmed
the in-repo surface (release notes draft, Firebase distribution wiring,
data-safety form) is GREEN; the items below are the external steps that
gate Phase F kickoff and cannot be verified by code review alone.

## Pre-kickoff checklist

### 1. BetaTesting platform brief

- [ ] Test brief uploaded to BetaTesting platform
- [ ] Brief framing matches `docs/release-notes/v2.0.0-draft.md` (Pro
      tier feature surface, wellness-aware productivity layer)
- [ ] Estimated tester time (~30 min daily for 3 weeks) called out in
      the brief so tester recruitment self-selects appropriately
- [ ] Phase F's 8-item checklist (medication, habits, tasks/AI,
      cross-device sync, batch ops, Pomodoro+, weekly review, settings/AI
      gate) included in the brief as the testing surface

### 2. Daily check-in mechanism

- [ ] Daily check-in mechanism configured on BetaTesting platform
- [ ] Check-in cadence calibrated so testers don't burn out
- [ ] Check-in prompts cover: bugs found, surfaces tested, daily-use
      habit (does the app stick?), one improvement suggestion

### 3. Opt-in link verification

- [ ] Opt-in link generated by BetaTesting platform
- [ ] Opt-in link tested on a fresh device / incognito browser to
      confirm flow works end-to-end
- [ ] Link copied to PrismTask's distribution channels (TestersCommunity
      community, Discord if applicable)

### 4. Build distribution

- [ ] Firebase tester group `phase-f-testers` created (or reuse existing
      tester group with renamed scope)
- [ ] `version-bump-and-distribute.yml` workflow's `appDistributionGroups`
      set to include `phase-f-testers` (verify in
      `app/build.gradle.kts` + workflow YAML)
- [ ] First test build distributed to the new group on a non-merge
      manual trigger (`gh workflow run`) to confirm distribution lands
      before kickoff
- [ ] Firebase App Distribution invite emails landing in tester inboxes
      (not spam)

### 5. Tester recruitment criteria

- [ ] Recruitment criteria reflect the brief's framing:
  - Active wellness-app users (not casual)
  - Mixed Android device population (Pixel + Samsung + OnePlus, API
    26-34)
  - At least 30% with neurodivergence-friendly feature interest (ADHD,
    autism — for the Brain Mode / UI Complexity / Forgiveness Streak
    surfaces)
  - At least 30% medication-tracking users (highest-engineered
    Phase F surface)
- [ ] Recruitment cap set so the 3-week round doesn't exceed bandwidth
      for triage (recommend 25-50 testers based on Avery's solo triage
      capacity)

### 6. Triage workflow ready

- [ ] `docs/launch/BUG_TRIAGE_TEMPLATE.md` reviewed and adopted as the
      triage rubric
- [ ] Weekly triage cadence calendared (recommend end-of-week
      Friday/Saturday review)
- [ ] Bug-tracking destination chosen (GitHub issues with `phase-f`
      label vs. BetaTesting platform's built-in bug intake — pick one
      to avoid duplicate tracking)

### 7. Pre-kickoff regression sweep

- [ ] `docs/release/PRE_PHASE_F_REGRESSION_SWEEP.md` methodology
      executed on the v1.7.X release artifact installed on S25 Ultra
- [ ] Go/no-go decision recorded against the methodology's rubric
- [ ] If NO-GO, blocker bugs filed and addressed before kickoff

### 8. Communication artifacts

- [ ] v2.0 release notes (Play Store copy + GitHub Release notes from
      `docs/release-notes/v2.0.0-draft.md`) editorial pass complete
- [ ] BetaTesting platform briefing message drafted for tester onboarding
- [ ] Day-1 announcement message drafted for TestersCommunity / other
      channels

## Day-of-kickoff checklist (2026-05-15)

- [ ] Manual `gh workflow run version-bump-and-distribute.yml` to push
      the kickoff build (do NOT rely on a coincidental same-day merge)
- [ ] Confirm first wave of testers (~5-10) actually receive and install
      the build before broader announcement
- [ ] Day-1 announcement sent
- [ ] Triage queue opened for first round of feedback (within 24h)

## Post-kickoff verification (week 1)

- [ ] Daily check-in response rate tracked (target: ≥ 60% of recruited
      testers responding daily by end of week 1)
- [ ] Bug-log triage executing on cadence (per `BUG_TRIAGE_TEMPLATE.md`)
- [ ] No silent install failures (Firebase distribution console shows
      expected install count vs. invite count)

## Stop-the-line conditions

If any of these surface in week 1, kickoff escalates to STOP-THE-LINE:

- Medication tracking surface has P0 data-loss bug → freeze new
  distributions, fix-and-redistribute pattern
- AI feature gate (privacy invariant) regression → freeze new
  distributions, treat as security incident
- Cross-device sync corruption → freeze new distributions, audit
  quarantine path
- Crash rate > 5% on cold start → freeze new distributions, triage
  Crashlytics

Each STOP-THE-LINE corresponds to a Phase F readiness scorecard surface
that this audit and PR #823 tracked as RED or YELLOW. Track in
`PRE_PHASE_F_MEGA_AUDIT.md` Phase 4 if triggered.
